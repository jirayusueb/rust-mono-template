# Vertical slice + clean architecture + handler-level CQRS

The API is an architecture showcase for a monorepo starter template. The Todo feature demonstrates three patterns composing in Rust/Axum: vertical slice (feature-first module structure), ports & adapters (hexagonal — domain at center, trait ports, swappable adapters), and CQRS (command/query handler separation).

We chose **handler-level CQRS** — separate command and query handler structs in the application layer, sharing a single `TodoRepository` port. This provides the structural separation CQRS is for (different handler types for reads vs writes) without the operational complexity of separate read/write stores or event sourcing. For a todo app, dual stores would be theater.

A single `AppError` enum covers all layers rather than layering domain/app/presentation error types. One feature doesn't justify the type ceremony; split when a second feature arrives with distinct error domains.

IDs are UUID v7, generated by the domain at construction time so the aggregate is self-identifying before persistence. SQLite stores them as TEXT.

**Considered alternatives:** flat module structure (rejected — layers wouldn't be visible enough for a showcase); full CQRS with separate read/write models (rejected — operational complexity unjustified for a todo); single error type per layer (rejected — premature abstraction for one feature).
